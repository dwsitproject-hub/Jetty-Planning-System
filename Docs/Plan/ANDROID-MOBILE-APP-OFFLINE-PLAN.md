# Android Mobile App (Offline-Capable) — Implementation Plan

Status: **Draft for review** · Created 2026-07-06 · Target: Android phone + tablet

This plan describes how to ship an Android app for the Jetty Planning System (JPS)
that **reuses the existing React frontend and Node/Express backend**, works on both
phones and tablets, and keeps working **offline** at the jetty (login + operational
transactions), syncing to the server when connectivity returns.

It is written to satisfy four constraints from the request:

1. **Do not change existing working code.** The web app and backend behavior stay intact.
   Mobile is additive: a native shell + a thin platform-adapter layer + new offline modules.
2. **Reuse the same data source, backend, and frontend** as much as possible.
3. **Run on both phone and Android tablet** (responsive; the app already has ~68 `@media` rules).
4. **Offline mode** — operators can log in and record transactions with no signal; changes
   sync automatically once the connection improves.

---

## 1. Approach: Capacitor wrapper (recommended)

Wrap the **existing** `Frontend/` React + Vite build in [Capacitor](https://capacitorjs.com/),
which packages the web app inside a native Android WebView and produces a real, installable
**APK/AAB**. Native device capabilities (secure storage, network status, SQLite, camera,
push) are reached through Capacitor plugins.

**Why Capacitor over the alternatives**

| Option | Frontend reuse | Real APK | Effort | Verdict |
|---|---|---|---|---|
| **Capacitor** | ~100% | Yes | Low | ✅ Chosen |
| React Native | Logic only; **all UI rebuilt** | Yes | Very high | ✗ Violates reuse constraint |
| PWA only | 100% | No (home-screen web) | Low | ✗ Not a distributable "app"; WebView-less |

Capacitor also lets a **single codebase** serve web and mobile: the same components render
in the browser (unchanged) and in the app. Mobile-only behavior is isolated behind a
capability check so **no existing component is forked or rewritten**.

**What gets added (all additive, nothing removed):**

```
Frontend/
  android/                         # NEW — native Android project (generated by Capacitor)
  capacitor.config.ts              # NEW — app id, name, server config, plugins
  src/
    platform/                      # NEW — platform-adapter layer (web vs native)
      index.js                     #   isNative(), network status, secure storage facade
    offline/                       # NEW — offline engine
      db.js                        #   SQLite/IndexedDB schema + access
      cache.js                     #   read-through response cache
      outbox.js                    #   queued mutations
      sync.js                      #   replay + conflict handling
      registry.js                  #   per-endpoint offline policy
    api/client.js                  # MODIFIED (small, backwards-compatible) — see §5
    context/AuthContext.jsx        # MODIFIED (small) — Bearer + offline login
```

The web build is unaffected: on a normal browser, `isNative()` is `false`, the offline
engine stays dormant, and `client.js` behaves exactly as today.

## 3. Backend impact: **config only, no code changes**

The backend already accepts token auth, so mobile needs **no new endpoints and no code edits**:

- `getTokenFromRequest` (Backend/src/middleware/auth.js) reads `Authorization: Bearer <jwt>`
  **before** falling back to the `jps_at` cookie.
- Bearer-authenticated requests are **CSRF-exempt** (Backend/src/middleware/csrf.js), so the
  app doesn't need the `jps_xsrf` double-submit dance.
- Login can return the JWT in the response body when `AUTH_RETURN_TOKEN_BODY=true`
  (Backend/src/routes/auth.js) — the app reads the token from the body instead of the cookie.

**Backend changes required:** set two env values on the API for the mobile use case —
`AUTH_RETURN_TOKEN_BODY=true` and add the app origin to `CORS_ORIGIN`
(`https://localhost` / `capacitor://localhost` for the WebView). That's it. The web app keeps
using cookies + CSRF exactly as before because it simply doesn't send a Bearer header.

> Optional later hardening: a dedicated `/auth/refresh` and shorter token TTL. Not required for v1.

## 4. Offline scope (v1 = field transactions)

Making all ~38 screens offline is unnecessary and costly. v1 targets the **workflows an
operator performs at the jetty**, where signal is poor. Everything else remains online-only
with a clear "requires connection" state.

**Offline-capable (read cached + queue writes):**
- **Login** (offline re-entry with cached credentials — see §6)
- **Allocation & Berthing** — view schedule/schematic; log arrival/berthing; drag reschedule
- **At-Berth Executions** — record operation timestamps/activities
- **Clearance / Verification** — record checks
- Reference/master data needed by the above (ports, jetties, SIs, shipment plans, lookups):
  cached read-only so forms render offline

**Online-only in v1 (show "needs connection"):**
- Admin (users/roles/API keys), Master-data **editing**, Reporting exports (Excel/PDF),
  Approvals, and **Jetty Live** RTSP video (cannot be cached).

The offline policy is declared **per endpoint** in `offline/registry.js` so scope can grow
later without touching components:

```js
// registry.js (illustrative)
export const OFFLINE_POLICY = [
  { match: /^\/allocation\/overview/,      read: 'cache',  ttlMs: 12*3600e3 },
  { match: /^\/allocation\/arrival$/,      write: 'outbox', entity: 'arrival' },
  { match: /^\/operations\//,              read: 'cache',  write: 'outbox', entity: 'operation' },
  { match: /^\/ports/,                     read: 'cache',  ttlMs: 7*24*3600e3 },
  // default: online-only (no read cache, writes fail fast with a clear message)
]
```

## 5. Integration seam: `api/client.js`

All 19 API modules call the shared helpers in `Frontend/src/api/client.js`
(`apiGet/apiPost/apiPut/apiPatch/apiDelete/apiPostForm`). This is the **only** place we
change — a small, backwards-compatible wrapper — so **no feature module is edited**.

Behavior by platform:

- **Web (isNative false):** unchanged. Same `fetch`, cookies, CSRF, timeouts.
- **Native + online:** attach `Authorization: Bearer <token>` (from secure storage) instead
  of relying on cookies; on a successful GET whose path matches a `read: 'cache'` policy,
  write the response into the local cache.
- **Native + offline:**
  - **GET** with a cache policy → return the cached copy (with an `X-From-Cache` marker so UI
    can show a "showing offline data" hint); otherwise throw a typed `OfflineUnavailableError`.
  - **POST/PUT/PATCH/DELETE** with a write policy → generate a client UUID, persist the request
    to the **outbox**, optimistically update the local cache, and return a synthetic success so
    the existing UI flow proceeds. Non-offline mutations throw a clear "needs connection" error.

Because the return shapes match today's helpers, **components need no changes** to benefit.

## 6. Auth & offline login

**Online login (native):** POST `/auth/login` → read JWT from body → store token + user
profile in **secure storage** (`@capacitor/preferences` encrypted, or a secure-storage
plugin). Also store a **salted hash** (PBKDF2/scrypt) of the password to permit offline
re-entry.

**Offline login:** if there's no network, the entered password is verified against the
locally stored hash; on match, the app restores the last session (cached token + profile)
and enters read/queue mode. When connectivity returns, the app performs a real login in the
background to obtain a fresh token and reconcile.

**Security posture (explicit tradeoff):** offline login inherently requires an on-device
credential check. Mitigations: strong KDF with per-install salt, secure storage, configurable
offline-grace window (e.g., offline login allowed for N days since last real login), remote
wipe of cached data on logout, and never storing the plaintext password. This tradeoff is
called out for sign-off in §12.

**Token expiry offline:** if the cached JWT expires while offline, queued writes still record;
on reconnect the app re-authenticates (silent re-login) before replaying the outbox, so the
server always sees a valid token.

## 7. Local data model (IndexedDB in v1)

> **v1 decision (P1):** use **IndexedDB** as the single offline store. It works
> identically in the browser and the Android WebView, needs no native plugin to
> debug on-device, and handles operational data volumes comfortably. The store is
> abstracted behind a small interface (`offline/store.js`), so switching to
> `@capacitor-community/sqlite` later — if data volume ever demands it — stays
> localized. Object stores mirror the logical tables below.

```
cache(
  key TEXT PRIMARY KEY,        -- normalized "METHOD path?query#portId"
  entity TEXT,                 -- e.g. 'allocation-overview'
  payload TEXT,                -- JSON response
  fetched_at INTEGER,          -- epoch ms
  ttl_ms INTEGER
)

outbox(
  id TEXT PRIMARY KEY,         -- client UUID
  created_at INTEGER,
  method TEXT, path TEXT, body TEXT,
  entity TEXT,                 -- for conflict/merge routing
  base_version TEXT,           -- server updated_at/etag captured at enqueue (for conflicts)
  status TEXT,                 -- 'pending' | 'sending' | 'failed' | 'done'
  attempts INTEGER, last_error TEXT
)

auth(
  id INTEGER PRIMARY KEY CHECK (id=1),
  token TEXT, profile TEXT,
  cred_hash TEXT, cred_salt TEXT,
  last_online_login_at INTEGER
)

meta(key TEXT PRIMARY KEY, value TEXT)   -- schema version, last sync, device id
```

## 8. Sync engine & conflict handling

**Trigger:** Capacitor `Network` `networkStatusChange` → online, plus app-resume and a periodic
timer. A mutex ensures one sync runs at a time.

**Replay:** send outbox rows oldest-first. On `2xx`, mark `done`, apply the server's canonical
response to the cache, and drop the optimistic copy. On `4xx` validation error, mark `failed`
and surface it for operator review (never silently drop). On network error, keep `pending`
and retry with backoff.

**Conflict policy (default = server-authoritative with surfaced conflicts):** at enqueue time
capture the record's `base_version` (server `updated_at`). On replay, send it as
`If-Unmodified-Since`/version guard where the endpoint supports it, or compare on return. If the
server changed since, **do not overwrite blindly** — mark the item `conflict` and show a
resolution card (keep mine / keep server / merge fields). Most JPS operational writes are
append-ish per-vessel timestamps, so true conflicts are rare, but they must be visible, not lost.

**Idempotency:** every queued write carries its client UUID; replays are safe to retry. (If we
later want server-side dedupe, the UUID is already there — but v1 needs no backend change.)

## 9. Platform-adapter layer (keeps components untouched)

`src/platform/index.js` centralizes the few environment differences so components stay generic:

- `isNative()` — Capacitor runtime check
- `getNetworkStatus()` / `onNetworkChange()` — Network plugin (web: `navigator.onLine`)
- `secureGet/secureSet` — Preferences/secure storage (web: `localStorage`, dev only)
- `pickFile()/takePhoto()` — Camera/Filesystem on native; `<input type=file>` on web
  (NOR/clearance/vessel-photo uploads)

Routing: switch `BrowserRouter` → `HashRouter` **only when `isNative()`** (WebView file origin
has no history server). One line in `main.jsx`, web unaffected.

## 10. Build & tooling

- **Config:** `capacitor.config.ts` (appId e.g. `com.energiup.jps`, appName "Jetty Planning",
  `webDir: dist`).
- **API URL:** mobile build sets an **absolute** `VITE_API_BASE_URL`
  (e.g. `https://jps.<domain>/api/v1`) — relative `/api/v1` can't work from a WebView origin.
- **Cleartext:** if staging is HTTP on a private IP, add an Android network-security-config
  allow-list (prod should be HTTPS).
- **Scripts (added to package.json):** `build:mobile`, `cap:sync`, `cap:open:android`,
  `apk:debug`. CI later: `assembleRelease` + signing.
- **Icons/splash:** generated from a provided logo; theme-color + manifest added to `index.html`
  (also upgrades the web app toward installable PWA at no extra cost).

## 12. Decisions — SIGNED OFF 2026-07-06

1. **Offline scope** — ✅ v1 = field transactions (Allocation, At-Berth, Clearance + reference
   reads). Broaden in a later version.
2. **Offline-login security** — ✅ Cache a strong credential hash on-device. Offline-grace window
   = **1 day**, and **configurable** (`OFFLINE_LOGIN_GRACE_DAYS`, default 1).
3. **Distribution** — ✅ **Sideloaded APK** for now; **Google Play (private/managed)** later.
   v1 signing = a stable release keystore suitable for both.
4. **Backend config** — ✅ Set `AUTH_RETURN_TOKEN_BODY=true` and extend `CORS_ORIGIN` for the app
   origin. No backend code change.
5. **Conflict UX** — ✅ **Server-authoritative + surfaced conflicts** (never silent last-write-wins;
   never silent-drop an operator entry). Chosen because operational writes must be auditable.

## 13. Risks & mitigations

- **WebView auth quirks** → solved by Bearer tokens (no cross-origin cookie reliance).
- **Silent data loss on sync** → outbox never auto-drops on validation/conflict; operator reviews.
- **Clock skew offline** → timestamps captured with device time + tz; server records receipt time;
  display both where it matters.
- **Large cache growth** → TTLs + LRU eviction + port-scoped partitioning.
- **File uploads offline** (NOR/photos) → store the file blob in the outbox and upload on sync.
- **Live video** → explicitly online-only; app shows a clear message offline.

## 14. Explicitly out of scope for v1
iOS build; full offline mirror of admin/master editing; offline report exports; push
notifications (can be a fast-follow); server-side idempotency/refresh-token endpoints.
